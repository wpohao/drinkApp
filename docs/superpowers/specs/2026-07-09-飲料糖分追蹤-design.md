# 飲料糖分追蹤 App — 設計文件（V1）

## 1. 背景與願景

長期目標是一個「個人健康 App」，包含三大支柱：**健身、飲食、情緒**，共用同一個帳號、UI 外殼與資料層。

本文件只聚焦 **飲食支柱的第一版（V1）**，切角為 **台灣手搖／飲料糖分追蹤**。這是刻意選的「小而在地」起點：範圍小、好上架、搜尋需求明確，之後能長成完整飲食支柱，再並行長出健身、情緒。

拆解原則：**做一個支援三支柱的殼，但 V1 對外只把「飲料糖分」這根柱子做完上架**，先驗證再擴張（lean MVP）。

## 2. 產品一句話

選一杯今天喝的飲料 → 自動算糖分與熱量 → 看今天還剩多少糖分額度、看歷史趨勢、必要時提醒與建議。

## 3. V1 範圍與非範圍

**做（V1）**
- 記錄一杯飲料（選店家／品項／糖度／容量／加料，或手動輸入）
- 自動計算糖分（g）與熱量（kcal）
- 今日糖分／熱量額度與剩餘量
- 歷史與統計（日／週／月趨勢、超標天數、最常喝品項）
- 內建常見手搖店品項資料庫（先收熱門幾家）＋查不到時手動補
- 每日糖分目標設定（預設 50g，可改 25g／自訂）
- 規則式提醒與建議（本地通知＋App 內模板建議，**非 AI**）
- AdMob 廣告

**不做（YAGNI，留給 V2+）**
- 帳號／登入、雲端同步
- AI（自然語言記錄、個人化建議）
- 健身、情緒支柱
- 社群、拍照辨識
- 食物／三餐（V1 只做飲料）
- 血糖記錄

## 5. 架構取向（方案 A：模組化，V1 對外只露飲食）

V1 對外是一支「專注做飲料糖分」的 App（App Store 故事單純、無半成品空 tab），但**程式內部分層模組化**，讓健身、情緒未來能當新模組掛入：

- **Core / Data 層**：SQLite 存取、資料模型、糖分計算邏輯、設定。與 UI 無關、可獨立測試。
- **Feature 模組**：`DrinkSugar`（V1 唯一啟用）。未來 `Fitness`、`Mood` 為同層的新模組。
- **共用 UI 層**：主題色、共用元件（環形進度、卡片、圖表包裝）、導覽外殼。
- **App 進入點**：V1 直接進入飲料糖分功能；未來擴充為多支柱導覽（tab／側欄）時，只需在外殼註冊新模組。

每個單元有單一職責、以明確介面溝通、可獨立理解與測試。

## 7. 糖分與熱量計算邏輯

每個品項在資料庫存「**全糖、基準容量**」下的糖分與熱量基準值，實際數值由糖度與容量組合算出：

```
size_ratio     = 所選容量.ml / 品項.base_size_ml
drink_sugar    = base_sugar_g × 糖度.multiplier × size_ratio
toppings_sugar = Σ ( topping.sugar_g × qty )          -- 加料為固定份量，不隨糖度／容量縮放
sugar_g        = drink_sugar + toppings_sugar
kcal           = ( base_kcal − base_sugar_g × (1 − 糖度.multiplier) × 4 ) × size_ratio
                 + Σ ( topping.kcal × qty )
```

- `× 4`：糖每公克約 4 kcal；減糖時扣掉未加入的糖熱量，奶／茶底的熱量不隨糖度變動。
- **加料（toppings）**：珍珠／椰果／布丁等每份有固定糖分與熱量，直接加總，不隨糖度或容量縮放。
- **糖度乘數**：無糖 0、一分糖 0.1、微糖（三分）0.3、半糖（五分）0.5、少糖（七分）0.7、八分糖 0.8、全糖 1.0（`sugar_level` 為資料表，可依店家增修）。
- **容量**：中杯（M）約 500ml、大杯（L）約 700ml、特大杯（XL）約 1000ml；`base_size_ml` 預設 700。
- `intake_log.sugar_g` 存**含加料的總糖分**（快照），加料明細另存於 `intake_log_topping`；計算結果在記錄當下落地存，日後修正基準值不影響歷史紀錄。

## 8. 內建資料庫策略（誠實面對資料問題）

台灣手搖店**大多不完整公開每杯糖克數**，因此「內建資料庫」實務上以**基準值 + 組合**方式維護：

- 每個品項只維護一組「全糖、基準容量」的糖分／熱量基準值。
  - 有官方公開數據 → 直接採用。
  - 無官方數據 → 依同類飲品合理估計，並標註為估計。
- 糖度×容量的其他組合由 §7 公式即時算出，**資料庫維護量小**。
- V1 先收錄熱門幾家（如 50嵐／清心福全／CoCo 都可／迷客夏／大苑子…）× 熱門品項（珍奶、紅／綠茶、奶茶、多多綠…）。實際清單於實作計畫階段整理（資料策展任務）。
- **加料**（珍珠／椰果／布丁／仙草…）以「單份糖分／熱量」基準維護，與品項基準值分離組合。
- **App 內數字統一標示為「參考估算」**，不宣稱醫療級精準；此定位在合規上也較安全。

## 9. 資料模型（SQLite 初稿）

```
drink_shop(
  id           INTEGER PK,
  name         TEXT NOT NULL,
  sort_order   INTEGER
)

drink_item(
  id           INTEGER PK,
  shop_id      INTEGER FK -> drink_shop.id,
  name         TEXT NOT NULL,
  category     TEXT,              -- 奶茶 / 純茶 / 果茶 ...
  base_sugar_g REAL NOT NULL,     -- 全糖、base_size_ml 下的糖分
  base_kcal    REAL NOT NULL,     -- 全糖、base_size_ml 下的熱量
  base_size_ml INTEGER NOT NULL,  -- 預設 700
  is_estimated INTEGER DEFAULT 1  -- 1=估計值 0=官方數據
)

sugar_level(
  code        TEXT PK,           -- none / s10 / s30 / half / s70 / s80 / full
  label       TEXT NOT NULL,     -- 無糖 / 一分糖 / 微糖 / 半糖 / 少糖 / 八分糖 / 全糖
  multiplier  REAL NOT NULL,     -- 0 / 0.1 / 0.3 / 0.5 / 0.7 / 0.8 / 1.0
  sort_order  INTEGER
)                                 -- 種子資料；各店家糖度命名不同，可增修

size_option(
  code        TEXT PK,           -- m / l / xl
  label       TEXT NOT NULL,     -- 中杯 / 大杯 / 特大杯
  ml          INTEGER NOT NULL   -- 500 / 700 / 1000
)

intake_log(
  id               INTEGER PK,
  logged_at        TEXT NOT NULL,   -- ISO8601 datetime
  item_id          INTEGER NULL FK -> drink_item.id,  -- 手動輸入時為 NULL
  custom_name      TEXT NULL,       -- 手動輸入的名稱
  sugar_g          REAL NOT NULL,   -- 含加料的總糖分，計算後快照
  kcal             REAL NULL,       -- 含加料的總熱量，計算後快照
  sugar_level_code TEXT NULL FK -> sugar_level.code,
  size_code        TEXT NULL FK -> size_option.code,
  note             TEXT NULL
)

topping(
  id           INTEGER PK,
  name         TEXT NOT NULL,     -- 珍珠 / 椰果 / 布丁 / 仙草 / 愛玉 ...
  sugar_g      REAL NOT NULL,     -- 單份糖分
  kcal         REAL NULL,         -- 單份熱量
  is_estimated INTEGER DEFAULT 1,
  sort_order   INTEGER
)

intake_log_topping(
  id            INTEGER PK,
  intake_log_id INTEGER FK -> intake_log.id,
  topping_id    INTEGER NULL FK -> topping.id,   -- 手動輸入時為 NULL
  custom_name   TEXT NULL,
  qty           INTEGER DEFAULT 1,
  sugar_g       REAL NOT NULL,    -- 單份×份量，計算後快照
  kcal          REAL NULL         -- 計算後快照
)

user_setting(
  id                  INTEGER PK CHECK(id=1),  -- 單列
  daily_sugar_target_g REAL NOT NULL DEFAULT 50,
  notify_threshold     INTEGER DEFAULT 1,  -- 接近/超標提醒開關
  notify_daily_log     INTEGER DEFAULT 0,  -- 每日補記提醒開關
  notify_daily_log_time TEXT NULL          -- 補記提醒時間
)
```

`sugar_level`、`size_option`、`topping` 為隨 App 內建的種子資料。

## 10. 提醒與建議（規則式，非 AI）

**提醒（iOS 本地通知，全部可於設定開關，預設不擾民）**
- 接近／超過每日糖分上限時提醒（例如達 80%、達 100%）。
- 可選每日補記提醒（使用者設定時間，例如晚上提醒記錄）。
- 需在首次使用時請求通知權限；使用者拒絕時功能靜默降級。

**建議（App 內、模板式，顯示於今日頁）**
- 依當日數據套用固定規則給出短建議，例如：
  - 超標 →「今天糖分超標囉，明天可以試試改點半糖或無糖。」
  - 連續數天達標 → 鼓勵訊息。
- 為固定文案集合，非 AI 生成。V2 再升級為 AI 個人化建議。

## 12. 產品路線圖

- **V1（本文件）**：飲料糖分追蹤 ＋ 規則式提醒／建議。上架驗證。
- **V2**：延伸食物／三餐糖分與營養；AI 建議（自然語言記錄、個人化）；無廣告＋雲端同步訂閱。
- **V3**：血糖記錄 —— **透過 Apple HealthKit 讀取血糖 ＋ 手動輸入，不做直接 BLE 配對**；與飲食／糖分交叉分析；定位為「記錄與參考」，避免醫療宣稱與醫療器材法規（TFDA／FDA）門檻。
- **並行**：健身、情緒各自長成新模組掛入同一個殼。

## 13. 合規注意

- 血糖相關功能（V3）維持「記錄／參考」定位，避免落入醫療器材規範。
- AdMob、通知等需符合 App Store 審核與隱私政策要求（含 App 隱私標籤）。
