## 1. 背景與願景

長期目標是一個「個人健康 App」，包含三大支柱：**健身、飲食、情緒**，共用同一個帳號、UI 外殼與資料層。

本文件只聚焦 **飲食支柱的第一版（V1）**，切角為 **台灣手搖／飲料糖分追蹤**。這是刻意選的「小而在地」起點：範圍小、好上架、搜尋需求明確，之後能長成完整飲食支柱，再並行長出健身、情緒。

拆解原則：**做一個支援三支柱的殼，但 V1 對外只把「飲料糖分」這根柱子做完上架**，先驗證再擴張（lean MVP）。

**平台選擇（2026-07-14 決策）**：V1 以 **Android 優先**。原因：開發者本機為 Windows，Android 可在 Windows 上用 Android Studio 完整開發／build／測試／上架，直接解除 iOS「必須有 Mac」的前置阻塞。iOS 改列為路線圖項目，待 Android 驗證後再移植（Core 邏輯已分層隔離，屆時主要是重寫 UI）。

## 2. 產品一句話

**控糖版手搖日記**：選一杯今天喝的飲料 → 自動算糖分與熱量 → 看今天還剩多少糖分額度、用日曆／圓餅圖回味與看趨勢，必要時提醒與建議。

> 與市面「手搖日記」類 App 的差異：它們多半只把甜度／冰塊當標籤記下來、**不計算糖克數、沒有每日控糖額度**。我們保留它們討喜的日記體驗，疊在「健康控糖核心」上——**算糖、每日額度、加料明細** 是我們的差異化護城河。

## 3. V1 範圍與非範圍

**做（V1）**
- 記錄一杯飲料（選店家／品項／糖度／容量／加料／**冰塊／星級評分**，或手動輸入）
- 自動計算糖分（g）與熱量（kcal）
- 今日糖分／熱量額度與剩餘量（環形進度）
- **歷史輸入快選**（記住常喝店家／品項，一鍵帶入，降低記錄摩擦）
- 歷史與統計：**日曆視圖（控糖顏色）**、**店家／品項圓餅圖（可點選篩選）**、日／週／月趨勢、超標天數、最常喝品項排行
- **月度戰績分享圖**（一鍵生成月報卡，分享到 IG 限動等）
- 內建常見手搖店品項資料庫（先收熱門幾家）＋查不到時手動補
- 每日糖分目標設定（預設 50g，可改 25g／自訂）
- 規則式提醒與建議（本地通知＋App 內模板建議，**非 AI**）
- AdMob 廣告

**不做（YAGNI，留給 V2+）**
- 帳號／登入、雲端同步
- AI（自然語言記錄、個人化建議）
- 健身、情緒支柱
- 好友／社群互動、留言（**月度戰績「分享圖」有做，但只是產圖＋系統分享，不含社群後端**）
- 拍照辨識
- 食物／三餐（V1 只做飲料）
- 血糖記錄
- iOS 版本（改列路線圖，Android 驗證後再移植）

## 4. 技術棧與限制

- **平台**：Android 8.0+（**minSdk 26**；涵蓋約 95%+ 現役裝置）。
- **語言／UI**：**Kotlin + Jetpack Compose**（宣告式 UI，對應原 SwiftUI）。
- **資料**：本機 **Room（SQLite）**，無 Server。
- **統計圖**：Compose 圖表——趨勢折線用 **Vico**；圓餅圖用支援 pie 的 Compose 圖表庫（如 compose-charts）或自繪 Canvas。
- **通知**：**NotificationManager + WorkManager**（排程每日補記提醒）；Android 13（API 33）以上需 `POST_NOTIFICATIONS` 執行期權限。
- **AI**：無（V1）。
- **變現**：AdMob（Google Mobile Ads Android SDK）。
- **相依管理**：**Gradle**（Kotlin DSL + Version Catalog `libs.versions.toml`）。
- **測試**：**JUnit4/5**（單元）＋ **Turbine**（測 `Flow`/`StateFlow`）；Room 用 in-memory database 測試。
- **開發工具**：**Android Studio（Windows 可）**——不需 Mac。
- **上架**：Google Play（開發者帳號一次性 $25 美元註冊費，可於 MVP 完成、準備上架時再申辦）。

> 相較 iOS 初版：解除了「iOS 開發／build／簽章／上架需要 Mac」的前置阻塞——Android 全程可在 Windows 完成。

## 5. 架構取向（方案 A：模組化，V1 對外只露飲食）

V1 對外是一支「專注做飲料糖分」的 App（商店故事單純、無半成品空 tab），但**程式內部分層**，讓健身、情緒未來能當新模組掛入。V1 用單一 `:app` Gradle 模組 + 分層 package（不過度切多模組，但保留未來拆模組空間）：

- **Core / Data 層**：Room 存取、資料模型、糖分計算邏輯、設定。與 UI 無關、可獨立測試。
  - `core/database`（Room：`@Entity`／`@Dao`／`@Database`、Migration、種子資料以 `RoomDatabase.Callback` 灌入）
  - `core/model`（Kotlin `data class`）
  - `core/logic`（`object SugarCalculator`，純函式）
  - `core/repository`（Kotlin `interface` + 實作，測試注入 fake）
- **Feature 模組**：`feature/today`、`feature/adddrink`、`feature/history`、`feature/settings`（各一組 `ViewModel` + Composable）。未來 `feature/fitness`、`feature/mood` 為同層的新模組。
- **共用 UI 層**：`ui`（主題色、共用 Composable：環形進度、卡片、圖表包裝、日曆）。
- **其他共用**：`advice`（規則式建議）、`notification`（本地通知）、`ads`（AdMob）。
- **App 進入點**：`MainActivity` + Compose Navigation。V1 直接進入飲料糖分功能；未來擴充為多支柱導覽（底部導覽／側欄）時，只需在外殼註冊新模組。

**MVVM**：`androidx.lifecycle.ViewModel` 持有 `StateFlow` 狀態（取代 SwiftUI 的 `@Published`），Composable 以 `collectAsStateWithLifecycle()` 訂閱；ViewModel 依賴 Repository 的 **interface**，測試時注入假物件。每個單元有單一職責、以明確介面溝通、可獨立理解與測試。

## 6. 功能與畫面（V1）

1. **今日（首頁）**
   - 今日已攝取糖分／熱量
   - 剩餘額度環形進度（對每日糖分目標；超標時環變紅）
   - 今天喝過的清單（可編輯／刪除，**刪除需二次確認防誤刪**）
   - 大顆「＋ 記一杯」主行動按鈕
   - App 內模板式建議（依當日數據顯示，見 §10）

2. **記一杯（新增）**
   - **歷史快選**：頂部以 chip 列出最近常喝的店家／品項，一鍵帶入（重用歷史紀錄，非另存表）。
   - 選店家 → 選品項 → 選糖度（無／一分／微／半／少／全…）→ 選容量（中／大／特大）→ 選加料（可多選＋份量，如珍珠／椰果／布丁）→ 選 **冰塊（正常／少／微／去冰／熱，記錄用，不影響算糖）** → **星級評分（1–5★，可留空）**
   - 自動帶出糖分／熱量預覽（含加料）→ 儲存
   - 找不到品項或加料 → 手動輸入名稱＋糖分（可選熱量）

3. **歷史／統計**
   - **日曆視圖**：月曆呈現哪天有紀錄，格子以控糖顏色標示（達標＝綠、超標＝紅、無紀錄＝灰）；點某天看當日明細。
   - **圓餅圖**：店家／品項占比統計；點某一塊可篩選出該店家（或品項）的歷史紀錄。
   - **趨勢**：日／週／月切換的糖分趨勢折線、超標天數統計、最常喝品項排行。
   - **月度戰績分享圖**：一鍵把當月統計（總杯數／最愛店家／控糖達標天數等）渲染成一張圖，透過系統分享（`ACTION_SEND`）分享到 IG 限動等。

4. **設定**
   - 每日糖分目標（預設 50g，可改 25g／自訂）
   - 通知偏好（各項可開關，見 §10）
   - 關於／版本／隱私政策
   - （V2 訂閱入口佔位，V1 不實作）

## 7. 糖分與熱量計算邏輯

每個品項在資料庫存「**全糖、基準容量**」下的糖分與熱量基準值，實際數值由糖度與容量組合算出（**與 iOS 初版完全相同，不因換平台而變**）：

```
size_ratio     = 所選容量.ml / 品項.base_size_ml
drink_sugar    = base_sugar_g × 糖度.multiplier × size_ratio
toppings_sugar = Σ ( topping.sugar_g × qty )          -- 加料為固定份量，不隨糖度／容量縮放
sugar_g        = drink_sugar + toppings_sugar
kcal           = ( base_kcal − base_sugar_g × (1 − 糖度.multiplier) × 4 ) × size_ratio
                 + Σ ( topping.kcal × qty )
```

- `× 4`：糖每公克約 4 kcal；減糖時扣掉未加入的糖熱量，奶／茶底的熱量不隨糖度變動。
- **加料（toppings）**：珍珠／椰果／布丁等每份有固定糖分與熱量，直接加總，不隨糖度或容量縮放。
- **糖度乘數**：無糖 0、一分糖 0.1、微糖（三分）0.3、半糖（五分）0.5、少糖（七分）0.7、八分糖 0.8、全糖 1.0（`sugar_level` 為資料表，可依店家增修）。
- **冰塊**：純記錄用，**不參與糖分／熱量計算**（無 multiplier）。
- **容量**：中杯（M）約 500ml、大杯（L）約 700ml、特大杯（XL）約 1000ml；`base_size_ml` 預設 700。
- `intake_log.sugar_g` 存**含加料的總糖分**（快照），加料明細另存於 `intake_log_topping`；計算結果在記錄當下落地存，日後修正基準值不影響歷史紀錄。

## 8. 內建資料庫策略（誠實面對資料問題）

台灣手搖店**大多不完整公開每杯糖克數**，因此「內建資料庫」實務上以**基準值 + 組合**方式維護：

- 每個品項只維護一組「全糖、基準容量」的糖分／熱量基準值。
  - 有官方公開數據 → 直接採用。
  - 無官方數據 → 依同類飲品合理估計，並標註為估計。
- 糖度×容量的其他組合由 §7 公式即時算出，**資料庫維護量小**。
- V1 先收錄熱門幾家（如 50嵐／清心福全／CoCo 都可／迷客夏／大苑子…）× 熱門品項（珍奶、紅／綠茶、奶茶、多多綠…）。實際清單於實作計畫階段整理（資料策展任務）。
- **加料**（珍珠／椰果／布丁／仙草…）以「單份糖分／熱量」基準維護，與品項基準值分離組合。
- **App 內數字統一標示為「參考估算」**，不宣稱醫療級精準；此定位在合規上也較安全。

## 9. 資料模型（Room / SQLite）

**表結構與 iOS 初版一致**，僅 `intake_log` 新增「冰塊、星級」兩欄，並新增 `ice_level` 種子表。欄位維持 snake_case（Room 以 `@ColumnInfo(name = ...)` 對應 Kotlin camelCase）。`logged_at` 維持 ISO8601 TEXT，讓 `date(logged_at,'localtime')` 的每日分組 SQL 照樣可用。

```
drink_shop(
  id           INTEGER PK,
  name         TEXT NOT NULL,
  sort_order   INTEGER
)

drink_item(
  id           INTEGER PK,
  shop_id      INTEGER FK -> drink_shop.id,
  name         TEXT NOT NULL,
  category     TEXT,              -- 奶茶 / 純茶 / 果茶 ...
  base_sugar_g REAL NOT NULL,     -- 全糖、base_size_ml 下的糖分
  base_kcal    REAL NOT NULL,     -- 全糖、base_size_ml 下的熱量
  base_size_ml INTEGER NOT NULL,  -- 預設 700
  is_estimated INTEGER DEFAULT 1  -- 1=估計值 0=官方數據
)

sugar_level(
  code        TEXT PK,           -- none / s10 / s30 / half / s70 / s80 / full
  label       TEXT NOT NULL,     -- 無糖 / 一分糖 / 微糖 / 半糖 / 少糖 / 八分糖 / 全糖
  multiplier  REAL NOT NULL,     -- 0 / 0.1 / 0.3 / 0.5 / 0.7 / 0.8 / 1.0
  sort_order  INTEGER
)                                 -- 種子資料；各店家糖度命名不同，可增修

ice_level(                        -- 【新增】冰塊，純記錄用，無 multiplier
  code        TEXT PK,           -- normal / less / light / free / hot
  label       TEXT NOT NULL,     -- 正常冰 / 少冰 / 微冰 / 去冰 / 熱
  sort_order  INTEGER
)                                 -- 種子資料

size_option(
  code        TEXT PK,           -- m / l / xl
  label       TEXT NOT NULL,     -- 中杯 / 大杯 / 特大杯
  ml          INTEGER NOT NULL   -- 500 / 700 / 1000
)

intake_log(
  id               INTEGER PK,
  logged_at        TEXT NOT NULL,   -- ISO8601 datetime
  item_id          INTEGER NULL FK -> drink_item.id,  -- 手動輸入時為 NULL
  custom_name      TEXT NULL,       -- 手動輸入的名稱
  shop_name        TEXT NULL,       -- 記錄當下店家名稱快照（供歷史快選／統計，不受品項刪除影響）
  sugar_g          REAL NOT NULL,   -- 含加料的總糖分，計算後快照
  kcal             REAL NULL,       -- 含加料的總熱量，計算後快照
  sugar_level_code TEXT NULL FK -> sugar_level.code,
  ice_level_code   TEXT NULL FK -> ice_level.code,    -- 【新增】冰塊
  size_code        TEXT NULL FK -> size_option.code,
  rating           INTEGER NULL,    -- 【新增】星級 1..5，可留空
  note             TEXT NULL
)

topping(
  id           INTEGER PK,
  name         TEXT NOT NULL,     -- 珍珠 / 椰果 / 布丁 / 仙草 / 愛玉 ...
  sugar_g      REAL NOT NULL,     -- 單份糖分
  kcal         REAL NULL,         -- 單份熱量
  is_estimated INTEGER DEFAULT 1,
  sort_order   INTEGER
)

intake_log_topping(
  id            INTEGER PK,
  intake_log_id INTEGER FK -> intake_log.id,
  topping_id    INTEGER NULL FK -> topping.id,   -- 手動輸入時為 NULL
  custom_name   TEXT NULL,
  qty           INTEGER DEFAULT 1,
  sugar_g       REAL NOT NULL,    -- 單份×份量，計算後快照
  kcal          REAL NULL         -- 計算後快照
)

user_setting(
  id                  INTEGER PK CHECK(id=1),  -- 單列
  daily_sugar_target_g REAL NOT NULL DEFAULT 50,
  notify_threshold     INTEGER DEFAULT 1,  -- 接近/超標提醒開關
  notify_daily_log     INTEGER DEFAULT 0,  -- 每日補記提醒開關
  notify_daily_log_time TEXT NULL          -- 補記提醒時間
)
```

- `sugar_level`、`ice_level`、`size_option`、`topping` 為隨 App 內建的種子資料。
- **歷史快選**不需新表：查 `intake_log` 最近的不重複 `shop_name`／品項即可（Repository 提供 `recentShops()`／`recentItems()`）。

## 10. 提醒與建議（規則式，非 AI）

**提醒（Android 本地通知，全部可於設定開關，預設不擾民）**
- 接近／超過每日糖分上限時提醒（例如達 80%、達 100%）。
- 可選每日補記提醒（使用者設定時間，例如晚上提醒記錄）——以 **WorkManager** 排程每日觸發。
- **Android 13（API 33）以上**需在首次使用時請求 `POST_NOTIFICATIONS` 執行期權限；使用者拒絕時功能靜默降級。

**建議（App 內、模板式，顯示於今日頁）**
- 依當日數據套用固定規則給出短建議，例如：
  - 超標 →「今天糖分超標囉，明天可以試試改點半糖或無糖。」
  - 連續數天達標 → 鼓勵訊息。
- 為固定文案集合，非 AI 生成。V2 再升級為 AI 個人化建議。

## 12. 產品路線圖

- **V1（本文件）**：Android 控糖手搖日記 ＋ 規則式提醒／建議。上架驗證。
- **V2**：延伸食物／三餐糖分與營養；AI 建議（自然語言記錄、個人化）；無廣告＋雲端同步訂閱（Google Play Billing）。
- **V3**：血糖記錄 —— **透過 Android Health Connect 讀取血糖 ＋ 手動輸入，不做直接 BLE 配對**；與飲食／糖分交叉分析；定位為「記錄與參考」，避免醫療宣稱與醫療器材法規（TFDA／FDA）門檻。
- **iOS 移植**：Android 驗證成功後，用 SwiftUI 重寫 UI 移植（Core 邏輯已分層隔離；屆時亦可評估 KMP 共用邏輯）。
- **並行**：健身、情緒各自長成新模組掛入同一個殼。

## 13. 合規注意

- 全 App 糖分／熱量數值定位為「參考估算」，不作醫療宣稱、不提供治療建議。
- 血糖相關功能（V3）維持「記錄／參考」定位，避免落入醫療器材規範。
- AdMob、通知等需符合 Google Play 審核與隱私政策要求（含 Play Data safety 表單、`POST_NOTIFICATIONS` 權限說明）。
- 資料全存本機、不上傳（對應「不收集任何資料」的隱私定位）。
